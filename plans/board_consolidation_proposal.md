# Board File Consolidation Proposal

**Date:** 2026-05-06  
**Scope:** Varmint 10X, Varmint 11X, and PixRacer Pro boards  
**Goal:** Reduce code duplication and improve maintainability

---

## Executive Summary

Analysis of the three board variants reveals **significant duplication** of STM32CubeMX-generated files. Approximately **85-90% of files are identical or nearly identical** across boards, representing ~2-3 MB of duplicated code per board.

**Key Finding:** The entire `Drivers/` directory tree (CMSIS + STM32 HAL) is duplicated across all three boards, despite being identical for STM32H7 family chips.

**Recommendation:** Consolidate common files into `boards/varmint_h7/common/` and use symbolic links or CMake configuration to reference them from board-specific directories.

---

## Redundancy Analysis

### 1. Completely Identical Files (Can be fully consolidated)

#### A. STM32 HAL Drivers (~2.5 MB per board)

**Location:** `Drivers/STM32H7xx_HAL_Driver/`

All three boards use identical STM32H7xx HAL driver files:

```
Drivers/STM32H7xx_HAL_Driver/
├── Inc/                          # ~80 header files (identical)
│   ├── stm32h7xx_hal_*.h
│   ├── stm32h7xx_ll_*.h
│   └── Legacy/
└── Src/                          # ~60 source files (identical)
    ├── stm32h7xx_hal_*.c
    └── stm32h7xx_ll_*.c
```

**Files:** 140+ files, ~2.5 MB each
**Duplication:** 3x copies = ~7.5 MB total
**Difference:** None - byte-for-byte identical

#### B. CMSIS Core (~500 KB per board)

**Location:** `Drivers/CMSIS/`

ARM CMSIS headers and device-specific files:

```
Drivers/CMSIS/
├── Include/                      # ARM Cortex-M core headers (identical)
│   ├── core_cm*.h
│   ├── cmsis_*.h
│   └── mpu_*.h
└── Device/ST/STM32H7xx/          # STM32H7 device headers
    ├── Include/
    │   ├── stm32h7xx.h           # Identical
    │   ├── stm32h753xx.h         # Varmint 10X/11X
    │   ├── stm32h743xx.h         # PixRacer Pro
    │   └── system_stm32h7xx.h    # Identical
    └── Source/Templates/
```

**Files:** 30+ files, ~500 KB each
**Duplication:** 3x copies = ~1.5 MB total
**Difference:** Only `stm32h753xx.h` vs `stm32h743xx.h` (chip variant)

#### C. System Call Stubs

**Location:** `Core/Src/syscalls.c`, `Core/Src/sysmem.c`

Newlib system call stubs for embedded systems:

```c
// syscalls.c - Identical across all boards (except copyright year)
int _getpid(void) { return 1; }
int _kill(int pid, int sig) { return -1; }
void _exit(int status) { while(1); }
// ... etc
```

**Files:** 2 files per board, ~10 KB each
**Duplication:** 3x copies = ~30 KB total
**Difference:** Copyright year only (2020-2023 vs 2020-2024)

#### D. Startup Code

**Location:** `Core/Startup/startup_stm32h7*.s`

Assembly startup code for STM32H7:

```
Core/Startup/
├── startup_stm32h753vihx.s       # Varmint 10X/11X
└── startup_stm32h743iikx.s       # PixRacer Pro
```

**Files:** 1 file per board, ~20 KB each
**Duplication:** 2x copies (10X/11X share one)
**Difference:** Chip-specific vector table and memory layout

### 2. Nearly Identical Files (Board-specific configuration)

#### A. STM32CubeMX Generated Files

**Location:** `Core/Src/main.c`, `Core/Inc/main.h`, `Core/Src/stm32h7xx_it.c`

These files contain peripheral initialization code generated by STM32CubeMX:

```c
// main.c structure (similar across boards)
void SystemClock_Config(void);      // Clock tree - board-specific
void MX_GPIO_Init(void);            // GPIO pins - board-specific
void MX_SPI1_Init(void);            // Peripherals - mostly similar
void MX_USART1_UART_Init(void);     // UARTs - board-specific
// ... etc
```

**Differences:**
- Clock configuration (similar but not identical)
- GPIO pin assignments (board-specific)
- Peripheral selection (e.g., SPI3 vs SPI5, UART7 vs UART8)
- Number of peripherals enabled

**Similarity:** ~70-80% identical code structure

#### B. HAL Configuration

**Location:** `Core/Inc/stm32h7xx_hal_conf.h`

HAL module enable/disable configuration:

```c
#define HAL_ADC_MODULE_ENABLED
#define HAL_SPI_MODULE_ENABLED
// ... etc
```

**Differences:** Minor - which HAL modules are enabled
**Similarity:** ~95% identical

#### C. Interrupt Handlers

**Location:** `Core/Src/stm32h7xx_it.c`, `Core/Inc/stm32h7xx_it.h`

Interrupt service routines:

```c
void TIM1_UP_IRQHandler(void);
void DMA1_Stream0_IRQHandler(void);
// ... etc
```

**Differences:** Which interrupts are used (board-specific)
**Similarity:** ~80% identical structure

### 3. Board-Specific Files (Cannot be consolidated)

#### A. Board Configuration

**Location:** `specific/BoardConfig.h`

Already board-specific, defines sensor configuration:

```c
#define INTERFACE_LIST \
  Sbus rc_;                 /* All */ \
  Ubx gps_;                 /* All */ \
  Adis165xx imu0_;          /* Varmint 10-12X */ \
  Bmi088 imu1_;             /* Varmint 10-12X */ \
  // ... board-specific sensors
```

**Status:** ✅ Already properly separated

#### B. Board Implementation

**Location:** `specific/Varmint_Init.cpp`, `specific/Callbacks.cpp`

Board-specific initialization and callbacks.

**Status:** ✅ Already properly separated

#### C. STM32CubeMX Project Files

**Location:** `*.ioc`, `.cproject`, `.project`, `CMakeLists.txt`

IDE and build configuration files.

**Status:** ✅ Must remain board-specific

## Consolidation Strategy

### Phase 1: Move Common HAL/CMSIS to Shared Location

#### Step 1.1: Create Common Drivers Directory

```bash
boards/varmint_h7/common/
├── Drivers/
│   ├── STM32H7xx_HAL_Driver/     # Move from each board
│   │   ├── Inc/
│   │   └── Src/
│   └── CMSIS/                     # Move from each board
│       ├── Include/
│       └── Device/ST/STM32H7xx/
│           ├── Include/
│           │   ├── stm32h7xx.h
│           │   ├── stm32h753xx.h  # Keep both variants
│           │   ├── stm32h743xx.h
│           │   └── system_stm32h7xx.h
│           └── Source/Templates/
└── Core/                          # New: common system files
    ├── Src/
    │   ├── syscalls.c             # Consolidated version
    │   └── sysmem.c               # Consolidated version
    └── Startup/
        ├── startup_stm32h753vihx.s
        └── startup_stm32h743iikx.s
```

#### Step 1.2: Update Board CMakeLists.txt

Each board's `CMakeLists.txt` should reference common drivers:

```cmake
# boards/varmint_h7/varmint_11X/CMakeLists.txt

# Include common HAL drivers
set(COMMON_DRIVERS_PATH ${CMAKE_CURRENT_SOURCE_DIR}/../common/Drivers)
include_directories(
    ${COMMON_DRIVERS_PATH}/STM32H7xx_HAL_Driver/Inc
    ${COMMON_DRIVERS_PATH}/CMSIS/Include
    ${COMMON_DRIVERS_PATH}/CMSIS/Device/ST/STM32H7xx/Include
)

# Add common HAL sources
file(GLOB_RECURSE COMMON_HAL_SOURCES
    "${COMMON_DRIVERS_PATH}/STM32H7xx_HAL_Driver/Src/*.c"
)

# Add common system files
set(COMMON_CORE_PATH ${CMAKE_CURRENT_SOURCE_DIR}/../common/Core)
set(COMMON_SOURCES
    ${COMMON_CORE_PATH}/Src/syscalls.c
    ${COMMON_CORE_PATH}/Src/sysmem.c
)

# Board-specific startup file
if(BOARD_VARIANT STREQUAL "H753")
    set(STARTUP_FILE ${COMMON_CORE_PATH}/Startup/startup_stm32h753vihx.s)
elseif(BOARD_VARIANT STREQUAL "H743")
    set(STARTUP_FILE ${COMMON_CORE_PATH}/Startup/startup_stm32h743iikx.s)
endif()

# Combine sources
set(BOARD_SOURCES
    ${COMMON_HAL_SOURCES}
    ${COMMON_SOURCES}
    ${STARTUP_FILE}
    # Board-specific files
    Core/Src/main.c
    Core/Src/stm32h7xx_it.c
    Core/Src/stm32h7xx_hal_msp.c
    Core/Src/system_stm32h7xx.c
)
```

#### Step 1.3: Remove Duplicated Directories

After verifying builds work:

```bash
# For each board
cd boards/varmint_h7/varmint_11X/
rm -rf Drivers/STM32H7xx_HAL_Driver
rm -rf Drivers/CMSIS
rm Core/Src/syscalls.c
rm Core/Src/sysmem.c
rm Core/Startup/startup_*.s
```

**Space Savings:** ~9 MB (3 boards × 3 MB each)

**Effort:** 30 minutes
**Risk:** Very Low
**Testing:** Verify .ioc files open correctly in STM32CubeMX

**Steps:**
1. Backup all three `.ioc` files
2. Modify `.ioc` files to disable HAL library copy:
   ```bash
   # For each board (varmint_10X, varmint_11X, pixracer_pro)
   cd boards/varmint_h7/<board_name>/
   
   # Backup
   cp <board_name>.ioc <board_name>.ioc.backup
   
   # Modify settings
   sed -i 's/ProjectManager.LibraryCopy=1/ProjectManager.LibraryCopy=0/' <board_name>.ioc
   sed -i 's/ProjectManager.DefaultFWLocation=true/ProjectManager.DefaultFWLocation=false/' <board_name>.ioc
   sed -i 's/ProjectManager.CustomerFirmwarePackage=/ProjectManager.CustomerFirmwarePackage=..\/common/' <board_name>.ioc
   ```
3. Test opening each `.ioc` file in STM32CubeMX
4. Verify CubeMX doesn't regenerate HAL drivers
5. Commit `.ioc` changes

**Rollback:** Restore from `.ioc.backup` files

### Phase 1: Consolidate HAL/CMSIS (Low Risk)

**Effort:** 2-4 hours
**Risk:** Low - these files are never modified
**Testing:** Verify all three boards compile and link

**Steps:**

2. Copy HAL/CMSIS from one board (they're identical)

## STM32CubeMX Compatibility

### Critical Issue: HAL Driver Regeneration

**Problem:** After consolidating HAL drivers to `common/Drivers/`, STM32CubeMX will attempt to regenerate them in the board-specific `Drivers/` directory when the `.ioc` file is opened or code is regenerated. This would undo the consolidation work.

### Current .ioc Configuration

All three boards currently have these settings in their `.ioc` files:

```ini
ProjectManager.DefaultFWLocation=true
ProjectManager.LibraryCopy=1
ProjectManager.FirmwarePackage=STM32Cube FW_H7 V1.11.1
ProjectManager.TargetToolchain=STM32CubeIDE
ProjectManager.UnderRoot=true
```

**Key Parameters:**
- `ProjectManager.LibraryCopy=1` - Copies HAL drivers into project
- `ProjectManager.DefaultFWLocation=true` - Uses default firmware location
- `ProjectManager.UnderRoot=true` - Generates files under project root

### Solution Options

#### Option 1: Disable Library Copy (Recommended)

Modify the `.ioc` files to reference HAL drivers from STM32Cube repository instead of copying them:

**Changes to `.ioc` files:**

```ini
# Change from:
ProjectManager.LibraryCopy=1
ProjectManager.DefaultFWLocation=true
ProjectManager.CustomerFirmwarePackage=

# To:
ProjectManager.LibraryCopy=0
ProjectManager.DefaultFWLocation=false
ProjectManager.CustomerFirmwarePackage=../common
```

**Pros:**
- CubeMX won't regenerate HAL drivers
- Drivers stay in `common/` directory
- Clean separation of generated vs common code

**Cons:**
- CubeMX may show warnings about missing drivers
- Need to ensure STM32Cube firmware package is installed
- Requires manual .ioc file editing

**Implementation:**

```bash
# For each board
cd boards/varmint_h7/varmint_11X/

# Edit varmint_11X.ioc
sed -i 's/ProjectManager.LibraryCopy=1/ProjectManager.LibraryCopy=0/' varmint_11X.ioc
sed -i 's/ProjectManager.DefaultFWLocation=true/ProjectManager.DefaultFWLocation=false/' varmint_11X.ioc
sed -i 's/ProjectManager.CustomerFirmwarePackage=/ProjectManager.CustomerFirmwarePackage=..\/common/' varmint_11X.ioc
```

#### Option 2: Use Symbolic Links

Create symbolic links from board directories to common drivers:

```bash
cd boards/varmint_h7/varmint_11X/
mkdir -p Drivers
ln -s ../../common/Drivers/STM32H7xx_HAL_Driver Drivers/STM32H7xx_HAL_Driver
ln -s ../../common/Drivers/CMSIS Drivers/CMSIS
```

**Pros:**
- CubeMX sees drivers in expected location
- No .ioc file changes needed
- Works with existing configuration

**Cons:**
- Symbolic links don't work well on Windows
- Git may have issues with symlinks
- CubeMX might still try to regenerate files
- Confusing directory structure

**Not recommended for cross-platform projects**

#### Option 3: Post-Generation Script

Keep current .ioc configuration but use a script to clean up after CubeMX regeneration:

```bash
#!/bin/bash
# post_cubemx_cleanup.sh

BOARD_DIR=$1  # e.g., boards/varmint_h7/varmint_11X

echo "Cleaning up CubeMX-generated HAL drivers for $BOARD_DIR..."

# Remove regenerated HAL drivers
rm -rf "$BOARD_DIR/Drivers/STM32H7xx_HAL_Driver"
rm -rf "$BOARD_DIR/Drivers/CMSIS"

# Keep only board-specific files in Drivers/ if any
# (Currently there are none)

echo "Cleanup complete. HAL drivers remain in common/"
```

**Pros:**
- No .ioc file changes
- Works with standard CubeMX workflow
- Easy to automate

**Cons:**
- Extra step after every CubeMX regeneration
- Risk of forgetting to run script
- Wasteful (generates then deletes)

#### Option 4: Hybrid Approach (Recommended for Teams)

Combine Option 1 with clear documentation and workflow:

1. **Modify .ioc files** to disable library copy (Option 1)
2. **Document CubeMX workflow** clearly
3. **Add build-time validation** to detect if HAL was regenerated
4. **Provide recovery script** if someone accidentally regenerates

**Implementation:**

```cmake
# Add to CMakeLists.txt
if(EXISTS "${CMAKE_CURRENT_SOURCE_DIR}/Drivers/STM32H7xx_HAL_Driver")
    message(FATAL_ERROR
        "ERROR: HAL drivers found in board directory!\n"
        "HAL drivers should be in common/Drivers/\n"
        "Run: rm -rf ${CMAKE_CURRENT_SOURCE_DIR}/Drivers/STM32H7xx_HAL_Driver\n"
        "This may have been caused by STM32CubeMX regeneration."
    )
endif()
```

### Recommended Workflow After Consolidation

#### For Developers Using STM32CubeMX:

1. **Before opening .ioc file:**
   - Commit all changes
   - Note current state of `Drivers/` directory

2. **After CubeMX regeneration:**
   - Check if `Drivers/STM32H7xx_HAL_Driver/` was recreated
   - If yes, delete it: `rm -rf Drivers/STM32H7xx_HAL_Driver Drivers/CMSIS`
   - Verify build still works
   - Commit only the intended changes (main.c, etc.)

3. **Best Practice:**
   - Use CubeMX sparingly after initial setup
   - Prefer manual peripheral configuration in code
   - Document any CubeMX changes in commit messages

#### Files Safe to Regenerate:

These files are board-specific and can be regenerated by CubeMX:

```
Core/Src/main.c                    # Use USER CODE sections
Core/Src/stm32h7xx_it.c            # Interrupt handlers
Core/Src/stm32h7xx_hal_msp.c       # MSP initialization
Core/Inc/main.h                    # Pin definitions
Core/Inc/stm32h7xx_it.h            # Interrupt declarations
```

## Risks and Mitigation

### Risk 1: STM32CubeMX Regeneration

**Problem:** STM32CubeMX may regenerate HAL drivers despite configuration changes

**Mitigation:**
- Modify .ioc files to disable library copy (see STM32CubeMX Compatibility section)
- Add build-time validation to detect regenerated drivers
- Document CubeMX workflow clearly
- Provide cleanup scripts for accidental regeneration
- Keep STM32CubeMX-generated files in board directories
- Use `/* USER CODE BEGIN */` sections for custom code

### Risk 2: Build System Complexity

**Problem:** More complex CMake configuration

**Mitigation:**
- Document CMake structure clearly
- Provide build examples in README
- Test builds on clean systems
- Keep fallback to original structure

### Risk 3: Debugging Difficulty

**Problem:** Harder to debug with shared code

**Mitigation:**
- Maintain clear file organization
- Use descriptive function names
- Add logging/tracing in common code
- Keep board-specific code in board directories

### Risk 4: Merge Conflicts

**Problem:** Multiple developers working on boards

**Mitigation:**
- Clear ownership of common vs board files
- Frequent integration
- Good commit messages
- Code review process

### Cleanup After Accidental CubeMX Regeneration

```bash
#!/bin/bash
# cleanup_cubemx_regen.sh
# Run this if CubeMX regenerates HAL drivers

BOARD_DIR=$1  # e.g., boards/varmint_h7/varmint_11X

if [ -d "$BOARD_DIR/Drivers/STM32H7xx_HAL_Driver" ]; then
    echo "Removing regenerated HAL drivers from $BOARD_DIR..."
    rm -rf "$BOARD_DIR/Drivers/STM32H7xx_HAL_Driver"
    rm -rf "$BOARD_DIR/Drivers/CMSIS"
    echo "Cleanup complete. HAL drivers remain in common/"
else
    echo "No regenerated drivers found in $BOARD_DIR"
fi
```
