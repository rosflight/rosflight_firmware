**Date:** 2026-05-06  
**Phase:** Phase 0 - Prepare for Consolidation

## Overview

Phase 0 modifies the STM32CubeMX `.ioc` project files to prevent automatic regeneration of HAL drivers after consolidation. This ensures that when HAL drivers are moved to `common/Drivers/`, STM32CubeMX will reference them instead of copying them into board-specific directories.

---

## Changes Made

### Modified Files

All three board `.ioc` files were modified:

1. [`boards/varmint_h7/varmint_10X/varmint_10X.ioc`](../boards/varmint_h7/varmint_10X/varmint_10X.ioc)
2. [`boards/varmint_h7/varmint_11X/varmint_11X.ioc`](../boards/varmint_h7/varmint_11X/varmint_11X.ioc)
3. [`boards/varmint_h7/pixracer_pro/pixracer_pro.ioc`](../boards/varmint_h7/pixracer_pro/pixracer_pro.ioc)

### Configuration Changes

For each `.ioc` file, three parameters were modified:

| Parameter | Before | After | Purpose |
|-----------|--------|-------|---------|
| `ProjectManager.LibraryCopy` | `1` | `2` | Reference HAL drivers instead of copying |
| `ProjectManager.DefaultFWLocation` | `true` | `false` | Use custom firmware location |
| `ProjectManager.CustomerFirmwarePackage` | _(empty)_ | `../common` | Point to common drivers directory |

**STM32CubeMX UI Mapping:**
- `LibraryCopy=1` → "Copy only the necessary library files"
- `LibraryCopy=2` → "Add necessary library files as reference in the toolchain project configuration file" ✅

### Verification

All changes verified successfully:

```

ProjectManager.DefaultFWLocation=false
ProjectManager.LibraryCopy=2

=== Varmint 11X ===

ProjectManager.DefaultFWLocation=false
ProjectManager.LibraryCopy=2

=== PixRacer Pro ===
ProjectManager.CustomerFirmwarePackage=../common
ProjectManager.DefaultFWLocation=false
ProjectManager.LibraryCopy=2
```

## STM32CubeMX Behavior with LibraryCopy=2

### Expected Behavior (User Tested)

With `LibraryCopy=2`, STM32CubeMX generates:

**✅ Files Generated:**
- `.mxproject` - Project metadata with HAL driver references
- `Core/Inc/` directory with:
  - `main.h` - Pin definitions and function prototypes
  - `stm32h7xx_hal_conf.h` - HAL configuration
  - `stm32h7xx_it.h` - Interrupt handler declarations
- `Core/Src/` directory with:
  - `main.c` - Peripheral initialization code
  - `stm32h7xx_hal_msp.c` - MSP initialization
  - `stm32h7xx_it.c` - Interrupt handlers

**❌ Files NOT Generated:**
- `Core/Startup/` directory - No startup files
- `Core/Src/syscalls.c` - Not generated
- `Core/Src/sysmem.c` - Not generated
- `Core/Src/system_stm32h7xx.c` - Not generated
- `Drivers/` directory - Not created (drivers referenced from `../common`)

**📝 Reference Mechanism:**
The `.mxproject` file contains references to HAL drivers in the custom location:
```
[PreviousLibFiles]
LibFiles=Drivers\STM32H7xx_HAL_Driver\Inc\stm32h7xx_hal_adc.h;...

[PreviousUsedCubeIDEFiles]
HeaderPath=Drivers\STM32H7xx_HAL_Driver\Inc;Drivers\CMSIS\Device\ST\STM32H7xx\Include;...
```

### Benefits of This Approach

1. **No Driver Duplication:** HAL drivers referenced from `../common`, not copied
2. **Minimal Code Generation:** Only board-specific files are generated
3. **Cleaner Structure:** System files (syscalls, sysmem, startup) not regenerated
4. **Perfect for Consolidation:** Exactly what we need for Phase 1
5. **IDE Integration:** References work with STM32CubeIDE toolchain

## Impact Analysis

### What This Changes

1. **STM32CubeMX Code Generation:**
   - ✅ Generates only `Core/Inc/` and `Core/Src/` files
   - ✅ Does NOT generate `Drivers/` directory
   - ✅ Does NOT generate `Core/Startup/` files
   - ✅ Does NOT generate `syscalls.c`, `sysmem.c`, `system_stm32h7xx.c`
   - ✅ References HAL drivers from `../common` via `.mxproject` file

2. **Files Still Generated by CubeMX:**
   - `Core/Src/main.c` - Peripheral initialization (with USER CODE sections)
   - `Core/Src/stm32h7xx_it.c` - Interrupt handlers
   - `Core/Src/stm32h7xx_hal_msp.c` - MSP initialization
   - `Core/Inc/main.h` - Pin definitions and prototypes
   - `Core/Inc/stm32h7xx_it.h` - Interrupt declarations
   - `Core/Inc/stm32h7xx_hal_conf.h` - HAL module configuration

3. **Files Now Managed Manually (Not Regenerated):**
   - `Core/Startup/startup_stm32h7*.s` - Startup code
   - `Core/Src/syscalls.c` - System call stubs
   - `Core/Src/sysmem.c` - Memory management stubs
   - `Core/Src/system_stm32h7xx.c` - System initialization
   - `Drivers/STM32H7xx_HAL_Driver/` - HAL drivers (will be in `common/`)
   - `Drivers/CMSIS/` - CMSIS headers (will be in `common/`)

### What This Does NOT Change (Yet)

- Build system (CMake) - No changes yet
- Directory structure - Drivers still in board directories
- Functionality - No changes to code behavior
- Existing builds - Still work as before

## Implications for Phase 1

This configuration is **perfect** for Phase 1 consolidation because:

1. **System Files Safe:** CubeMX won't regenerate `syscalls.c`, `sysmem.c`, or startup files
2. **No Driver Conflicts:** CubeMX won't create `Drivers/` directory
3. **Clean Separation:** Only board-specific peripheral config is regenerated
4. **Easier Consolidation:** Can move system files and drivers without CubeMX interference
5. **Reference-Based:** HAL drivers referenced from `../common` location

### Phase 1 Strategy

With this configuration, Phase 1 workflow:

1. **Move HAL/CMSIS to common** - CubeMX will reference them from `../common` ✅
2. **Move system files to common** - CubeMX won't recreate them ✅
3. **Move startup files to common** - CubeMX won't recreate them ✅
4. **Keep Core/ in boards** - CubeMX will continue to manage these ✅
5. **Update CMake** - Point to common drivers ✅

6. Build references common drivers via CMake
7. Build and test as normal

### What Developers Need to Know

**✅ Safe to Regenerate:**
- `Core/Src/main.c` - Use `/* USER CODE BEGIN */` sections for custom code
- `Core/Src/stm32h7xx_it.c` - Interrupt handlers
- `Core/Src/stm32h7xx_hal_msp.c` - MSP initialization
- `Core/Inc/*.h` - Header files

**⚠️ Do NOT Regenerate (Managed Manually):**
- `Core/Startup/` - Startup files (will be in `common/`)
- `Core/Src/syscalls.c` - System calls (will be in `common/`)
- `Core/Src/sysmem.c` - Memory management (will be in `common/`)
- `Core/Src/system_stm32h7xx.c` - System init (board-specific, keep in board)
- `Drivers/` - HAL drivers (will be in `common/`)

**Good News:** With `LibraryCopy=2`, CubeMX won't regenerate the "Do NOT Regenerate" files!
